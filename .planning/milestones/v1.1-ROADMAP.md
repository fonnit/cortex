## Goal

Restore backend isolation and make the ingest pipeline reliable. Daemon stops touching Neon, classification moves to queue-driven consumers, and `claude -p` switches from argv content to file paths. v1.1 is invisible to user-facing surfaces; it makes the pipeline reliable enough to run unattended.

## Why

**Trigger:** Daemon-to-Neon direct access violated backend isolation (any daemon bug could corrupt the DB). Claude CLI integration via execFile broke on binary files, argument size limits, and file-descriptor exhaustion. Gmail Stage 2 never ran (keep-emails stuck at `processing` permanently). Repo scanning hit EMFILE when traversing `.git` and `node_modules` directories.

## Decisions Made (locked during this milestone)

| Decision | Phase | Rationale |
|----------|-------|-----------|
| Daemon must not access Neon directly | 5/6 | Backend isolation; bug containment |
| `claude -p` receives file paths, never content as argv | 5/7 | Argv content broke on binary, EBADF, size limits |
| Queue-driven consumers replace inline scan classification | 5/7 | Fixes stuck `processing` items + unblocks Gmail Stage 2 |
| Retry/claim metadata in `classification_trace.queue` JSON | 5 | No schema migration needed (honours v1.1 non-goal) |
| `Item.status` extended additively (5 new strings) | 5 | Status is `String`, not enum; zero migration |
| 5-attempt retry cap → terminal `status='error'` | 5 | Bounded recovery; no infinite-bounce |
| 10-min stale-claim timeout | 5 | Long enough for `claude -p` on size-band-capped files |
| In-memory FIFO buffer cap 100 with overflow-drop-oldest | 6 | Enough for single-operator MVP; rediscovery is cheap |
| Single Node process for both consumer pools | 7 | Simpler launchd surface; pools independent via semaphores |
| `CORTEX_API_KEY` in plain env var via launchd plist | 6/7 | Auditable via `launchctl print`; Keychain deferred |
| Phase 8 toolkit (not live runs) is the deliverable | 8 | Live runs require operator-controlled cycles |
