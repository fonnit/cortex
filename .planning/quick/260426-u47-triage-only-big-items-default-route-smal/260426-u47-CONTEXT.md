# Quick Task 260426-u47: Triage only big items, default-route small to Stage 2 with auto-label/auto-ignore — Context

**Gathered:** 2026-04-26
**Status:** Ready for planning

<domain>
## Task Boundary

Restructure the consumer ingest pipeline so that small, low-risk items skip the Stage 1 relevance gate and go straight to Stage 2, where Stage 2 itself can produce a final action (auto-file or auto-ignore) without human triage when claude is confident enough.

In scope:
- Routing decision (Stage 1 vs straight-to-Stage 2) based on item size.
- Stage 2 schema and prompt changes to support an explicit `decision` field (auto_file / ignore / uncertain) alongside the existing 3-axis output.
- Consumer state machine + `/api/classify` route updates so Stage 2 can finalize an item (status moves to `filed` or `ignored`) without going through human review.
- Cold-start safety so auto-actions can't misfire before the taxonomy is bootstrapped.

Out of scope (handled in sibling quick tasks):
- Letting Stage 2 propose **new** labels (Task #2 — separate `/gsd-quick`).
- Triage UI selector styling (Task #3 — separate `/gsd-quick`).

</domain>

<decisions>
## Implementation Decisions

### Stage 1 routing threshold
- **Stage 1 (relevance gate) runs only when:**
  - source = `downloads` AND `size_bytes > 1_048_576` (1 MiB), OR
  - source = `gmail` AND has at least one attachment with `size_bytes > 1_048_576`.
- **All other items go straight to Stage 2:** small Downloads files, all emails with no attachments, all emails whose attachments are ≤ 1 MiB.
- **Implication:** Stage 1 becomes a defensive gate for content claude has to actually read (large files cost tokens). Small items, where the metadata block alone is enough signal, skip it.

### Auto-file confidence threshold
- **Auto-file when** Stage 2 returns axes with **all three** axes at confidence ≥ 0.85 **AND** all three axes have a non-null value **AND** all three values match labels that already exist in `TaxonomyLabel` (i.e., none of the axes returned a newly-proposed label).
- This is the same 0.85 threshold the prompt already uses for "confident match." We're now acting on it instead of just decorating the row.
- **Status transition for auto-filed items:** Stage 2 worker sets `confirmed_drive_path = proposed_drive_path` and moves status from `processing_stage2` to a "filed" terminal state (planner picks the exact name — likely `filed` or `auto_filed` to distinguish from human-confirmed).

### Auto-ignore semantics
- **Add an explicit `decision` field to the Stage 2 output schema:** `decision: 'auto_file' | 'ignore' | 'uncertain'`.
- **Stage 2 prompt** is updated to instruct claude that it may return `decision: 'ignore'` for junk that doesn't deserve a label (e.g., spam, marketing, automated security alerts).
- **Auto-ignore action:** when `decision === 'ignore'` and `confidence ≥ 0.85`, set status to `ignored` directly from Stage 2. Skip filing, skip Drive resolution.
- This is symmetric with the auto-file path: claude makes a confident terminal call, the system honors it without human review.

### Cold-start safety
- **Hard precondition for any auto-action** (auto-file OR auto-ignore on `decision: 'ignore'` with axes attempted):
  - All 3 axes have `value !== null`, AND
  - None of the 3 axis values is a newly-proposed label — every value must already exist in `TaxonomyLabel` for that axis.
- **Why this is enough:** with the current closed-vocab prompt (Task #1 ships before Task #2), claude can never return a value not in the lists, so the "not new" check is trivially satisfied — but it future-proofs against the Task #2 prompt change. Once Task #2 ships and claude can propose new labels, this guard automatically prevents auto-actions from acting on unverified vocabulary.
- **Auto-ignore exception:** if claude returns `decision: 'ignore'` it doesn't need to populate axes (we trust the ignore signal alone). Auto-ignore precondition is only the confidence threshold; cold-start guard does not apply because no labels are being committed.

### Claude's Discretion (planner picks)
- Exact name of the new terminal status (`filed` vs `auto_filed` vs `classified` — pick whatever reads cleanest in the triage UI).
- Whether `decision: 'uncertain'` and "low confidence + null axes" share a code path or remain distinct in the state machine.
- Whether the Stage 2 prompt's existing "Never invent labels" line stays (closed vocab) or comes out — leave it for now since Task #2 changes it separately.
- Schema: whether the `decision` field is sibling to `axes` or whether the schema becomes a discriminated union by `decision`.
- Where the "all 3 axes are existing labels" check runs (consumer worker vs `/api/classify` route). Recommend the route — it's the boundary that already validates Stage 2 payloads.

</decisions>

## Specific Ideas

- Threshold constant: `STAGE1_MIN_SIZE_BYTES = 1_048_576` (1 MiB exactly, not 1_000_000).
- Auto-action confidence threshold: `0.85` (matches existing `STAGE2_CONFIDENT_MATCH = 0.85` from the prompt).
- The routing decision lives in the daemon's ingest path or in `/api/queue` — not in the consumer worker. The consumer should receive items already tagged `pending_stage1` or `pending_stage2` and just process them. (Planner: confirm by reading `app/api/ingest/route.ts` and `app/api/queue/route.ts`.)
